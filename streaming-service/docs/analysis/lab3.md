# Аналіз лабораторної 3

## 1. Що змінилося порівняно з лабораторною 2

У лабораторній 2 проєкт був поділений на шари `presentation`, `application`, `domain` та `infrastructure`. У лабораторній 3 до цього додається розділення на **команди** та **запити**, тобто `application` окремо обробляє зміну стану і читання даних.

Практично це означає:

- контролери лише маплять HTTP-запит у `Command` або `Query`;
- для запису використовуються `CommandHandler`;
- для читання використовуються `QueryHandler`;
- `Command` описує зміну стану, а `Query` - отримання даних;
- `Query` повертає `Read Model` або DTO, а не доменну модель.

## 2. Переваги CQS

Головні переваги CQS - краща читабельність архітектури, менше побічних ефектів і чітке розділення читання та запису. Це спрощує підтримку коду, дозволяє окремо оптимізувати query-шлях і робить тестування команд простішим.

Додатково CQS зменшує зв’язок між API і доменом: `Query` може повертати дані у зручному для клієнта форматі, не змінюючи внутрішню модель.

## 3. Недоліки CQS

Основний недолік - збільшення кількості класів, DTO, handler-ів і мапінгу між шарами. Через це архітектура стає більша і потребує більше дисципліни.

Також зростає когнітивна складність: щоб зрозуміти одну операцію, часто потрібно пройти ланцюг controller → command/query → handler → domain/repository або read repository → response DTO.

## 4. Command / Query Handler vs Service

Звичайний Service часто поєднує читання, запис і оркестрацію в одному класі. Через це він стає “все-в-одному” і гірше масштабується.

Підхід із `CommandHandler` та `QueryHandler` дає чіткі ролі:

- `CommandHandler` змінює стан і працює через домен;
- `QueryHandler` лише читає дані та повертає read model;
- controller не містить бізнес-логіки.

Це зменшує ризик побічних змін: нова команда не вимагає переписувати великий сервіс, а зміна формату виводу зачіпає лише query-частину.

## 5. Як CQS впливає на розширюваність

CQS робить систему зручнішою для розширення: нова команда зазвичай додає новий DTO і handler, а існуючі запити не змінюються.

Це корисно, коли write-path і read-path мають різний життєвий цикл. У результаті розробники можуть паралельно працювати над записом і читанням, майже не заважаючи один одному.

## 6. Чим Query model відрізняється від доменної моделі

`Query` не повинен повертати доменну модель. Замість цього використовується **Read Model** або DTO, оптимізований під конкретний сценарій читання.

Це дає три основні переваги: менше зайвих даних у відповіді, стабільніший API та можливість мати різні read models для списку, деталей або адміністративного перегляду.

## 7. Тестування: що стало простіше, а що складніше

Порівняно з лабораторною 2, тестування стало більш спеціалізованим. Команди зручно перевіряти як **unit-тести** з mock- або fake-залежностями: чи виконується доменна логіка, чи відхиляється невалідний стан, чи повертається очікувана помилка.

Query-частину краще перевіряти як **integration-тести**, бо тут важливі SQL, фільтрація, проєкції та коректність read model. До цього підходять тести на кшталт `UserControllerIntegrationTest`, `MovieControllerIntegrationTest`, `ActorControllerIntegrationTest`, `SubscriptionPlanControllerIntegrationTest`.

Отже, команди тестувати простіше, а запити - складніше, але загалом це робить тестову стратегію точнішою.

## 8. Висновок

Лабораторна 3 продовжує ідеї лабораторної 2: до шарової архітектури додається розділення на **запис** і **читання**. CQS робить код структурованішим, покращує тестування і спрощує окремий розвиток read-path та write-path.

Для цього проєкту підхід корисний, бо дозволяє відокремити зміну стану від отримання даних і використовувати окремі моделі для команд та запитів без змішування їх із доменною логікою.
