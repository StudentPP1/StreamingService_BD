# Implemented Use Cases 

---

## UC1. Sign Up

**Актор:** Незареєстрований користувач  
**Передумови:** Користувач не має акаунту в системі  
**Основний сценарій:**
1. Користувач відкриває сторінку реєстрації
2. Система відображає Sign Up/Log in
3. Користувач обрав Sign Up
4. Система перенаправляє на UC2

**Альтернативні сценарії:**
- Користувач вже зареєстрований → система показує повідомлення та пропонує увійти (UC3)

---

## UC2. Sign Up Using Email and Password

**Актор:** Незареєстрований користувач  
**Передумови:** Користувач обрав реєстрацію через email  
**Основний сценарій:**
1. Користувач вводить ім'я, прізвище, email, пароль, дату народження
2. Система валідує дані (формат email, довжина та складність пароля)
3. Система перевіряє унікальність email
4. Система хешує пароль та зберігає користувача в БД з роллю `ROLE_USER`
5. Система створює сесію і повертає користувача як автентифікованого

**Альтернативні сценарії:**
- Email вже зареєстрований → `UserAlreadyExistsException`, повідомлення про конфлікт
- Невалідний формат email або слабкий пароль → повідомлення про помилку валідації
- Неповні дані → повідомлення про обов'язкові поля

**Основний сценарій:**

## UC4. Log In Using Email and Password

**Актор:** Зареєстрований користувач  
**Передумови:** Користувач зареєстрований через email  
**Основний сценарій:**
1. Користувач вводить email та пароль
2. Система передає дані в `AuthenticationManager`
3. `DaoAuthenticationProvider` знаходить користувача за email
4. Система перевіряє хеш пароля (BCrypt)
5. Система створює `Authentication` з роллю та зберігає у `SecurityContext`
6. Система створює HTTP-сесію (`JSESSIONID`)

**Альтернативні сценарії:**
- Невірний email або пароль → `BadCredentialsException`, повідомлення про помилку
- Акаунт не знайдено → `UsernameNotFoundException`
- Акаунт заблокований → повідомлення про блокування

## UC6. Make Payment Online by Card

**Актор:** Авторизований користувач  
**Передумови:** Користувач обрав план підписки  
**Основний сценарій:**
1. Користувач вводить дані картки
2. Система передає дані до платіжного провайдера
3. Провайдер підтверджує платіж
4. Система зберігає запис про платіж та активує підписку

**Альтернативні сценарії:**
- Платіж відхилено провайдером → повідомлення про помилку, підписка не активується
- Таймаут запиту до провайдера → повідомлення про технічну помилку, повторна спроба
- Недостатньо коштів → відповідне повідомлення від провайдера

## UC10. Cancel Subscription

**Актор:** Авторизований користувач  
**Передумови:** Користувач має активну підписку (`ACTIVE`)  
**Основний сценарій:**
1. Користувач обирає підписку та натискає «Скасувати»
2. Система отримує підписку з **песимістичним блокуванням** (`SELECT FOR UPDATE`)
3. Система перевіряє що підписка належить користувачу
4. Система перевіряє що статус `ACTIVE`
5. Система встановлює статус `CANCELLED` та зберігає

**Альтернативні сценарії:**
- Підписка вже скасована або прострочена → `SubscriptionNotActiveException`
- Підписка не належить користувачу → 403 Forbidden
- Одночасний запит на скасування → другий запит чекає на блокування, потім отримує `SubscriptionNotActiveException`

## UC11. Buy Subscription

**Актор:** Авторизований користувач  
**Передумови:** Користувач авторизований, доступні активні плани  
**Основний сценарій:**
1. Користувач обирає план підписки (UC12)
2. Система відображає форму оплати
3. Користувач оплачує (UC7)
4. Система створює запис `UserSubscription` зі статусом `ACTIVE`

**Альтернативні сценарії:**
- Обраний план було деактивовано адміном під час оформлення → повідомлення про недоступність плану
- Платіж не пройшов → підписка не створюється

## UC13. Admin Log In Using Email and Password

**Актор:** Адміністратор  
**Передумови:** Адмін має акаунт з роллю `ROLE_ADMIN`  
**Основний сценарій:**
1. Адмін відкриває сторінку входу
2. Вводить email та пароль
3. Система автентифікує аналогічно до UC5
4. Система перевіряє наявність ролі `ROLE_ADMIN`
5. Надається доступ до адмін-панелі

**Альтернативні сценарії:**
- Акаунт має лише роль `ROLE_USER` → доступ до адмін-панелі заборонено (403)
- Невірні credentials → повідомлення про помилку

## UC15. Manage Subscription Plan (Admin)

**Актор:** Адміністратор  
**Передумови:** Адмін авторизований  
**Основний сценарій:**
1. Адмін відкриває розділ управління планами
2. Адмін може створити новий план або редагувати існуючий (назва, ціна, опис)
3. Система зберігає зміни
4. При деактивації плану - система скасовує всі активні підписки на цей план (`cancelAllByPlan`)

**Альтернативні сценарії:**
- Невалідна ціна (від'ємна або нуль) → повідомлення про помилку валідації
- Деактивація плану з активними підписками → система автоматично переводить їх у `CANCELLED`
